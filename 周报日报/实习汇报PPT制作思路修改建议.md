# 实习汇报 PPT 制作思路修改建议

参考文件：

- `工作日报汇总报告.md`
- `实习汇报ppt制作思路.docx`

## 一、总体判断

当前 `实习汇报ppt制作思路.docx` 的整体流程是可行的，主线为：

工作内容展示 -> 方法论成长 -> 踩坑与解决 -> 团队协作 -> 学习方法 -> 不足规划 -> 总结致谢

这个结构适合实习汇报，因为它不只是罗列做了什么，也能体现“从学生到游戏客户端程序”的成长过程。

但从日报总结来看，当前方案需要明显调整。主要问题是：部分内容过于通用，和实际工作经历匹配度不够；有些表述容易显得夸大；有些重要成果没有被放到核心位置。

## 二、当前方案的主要问题

### 1. “工作内容展示”需要更具体

原文中写到“参与 XX 款游戏/模块”“完成 XX 个功能点”，这是占位表达，后续必须替换为真实内容。

建议改成：

- 参与热更框架验证 Demo、类“元气骑士”移植 Demo、类“是功夫”项目开发。
- 覆盖战斗系统、技能系统、UI 系统、局外养成、配置表、资源接入、多端打包、性能优化等模块。
- 可展示实际成果数量，例如：52 份日报、3 个阶段、多个局内外系统、微信小游戏/安卓/iOS 打包测试经验。

这一部分最好用时间线展示，而不是单纯堆截图。

推荐页结构：

1. 实习整体时间线：学习适应期、Demo 验证期、项目开发期、打包优化收尾期。
2. 工作模块总览：战斗、UI、技能、配置表、打包、性能优化。
3. 代表性成果截图：类“元气骑士”Demo、类“是功夫”战斗界面、秘籍/勾玉/兵器等局外系统。

### 2. “工业级开发思维”可以保留，但标题和案例要落地

原文中的“反脆弱设计”表达偏大，作为实习汇报容易显得概念化。日报中真实体现的是：

- 从能跑通功能，到考虑异常数据、资源缺失、平台差异、UI 层级、异步时序。
- 从直接写逻辑，到使用配置表、enum、通用 UI、状态管理降低维护成本。
- 从碰撞检测、单个动画逻辑，到距离检测、帧动画、批处理、资源压缩等优化思路。

建议把标题改为：

- 从“功能实现”到“可维护开发”
- 从“能跑通”到“能上线测试”
- 从“单点修 Bug”到“建立排查流程”

对应案例建议使用日报里的真实内容：

- 配置表字段和代码字段不一致导致读取失败，后续通过字段规范、enum 管理减少风险。
- UI Button 点不到、点击穿透、Canvas 层级混乱，后续总结 Raycaster/Image/Canvas 使用规则。
- await、TryWait、out 参数在 ILRuntime/热更环境下出现时序或传值问题，后续形成异步写法约束。
- 微信小游戏、安卓、iOS 打包问题，后续形成打包检查清单。

### 3. “组件化抽象”不要写成已经有完整组件库

原文写“将通用按钮、弹窗、列表抽象成基础组件库”“开发时间缩短约 30%”，这个说法如果没有实际数据支撑，建议不要使用。

日报中真实发生的是：

- 制作通用弹窗 UI。
- 复用技能/武器展示框。
- 通过通用 UI 预制体减少相似界面的制作成本。
- 对空白点击关闭、详情弹窗、列表展示等 UI 形成复用思路。

建议改成：

“在练功房、属性详情、秘籍、兵器、奖励弹窗等界面开发中，逐步意识到通用 UI 预制体和统一交互规则的重要性，开始将重复的弹窗、图标框、详情面板做成可复用结构，减少后续界面的重复制作和层级问题。”

### 4. “Profiler 定位卡顿”需要确认是否真实使用

原文写“学会使用 Profiler 定位内存泄漏或 DrawCall 过高问题”。日报中确实有性能优化、GC、DrawCall、Overdraw、Shader、贴图、动画、碰撞等总结，但没有明确写使用 Profiler。

如果汇报时没有实际 Profiler 截图或数据，不建议重点写“使用 Profiler 定位”。可以改成：

- 通过敌人数量压力测试观察性能瓶颈。
- 分析 GC、碰撞、动画、资源大小、Shader、UI 叠层等可能影响性能的因素。
- 对部分逻辑做距离检测替换、帧动画转换、图片压缩、资源预热和批处理思路总结。

如果后续能补一张 Profiler 截图，再把它作为性能优化案例会更有说服力。

### 5. “踩坑实录”非常适合，但例子要换成真实坑

原文中的多端适配例子是可用方向，但“抽象适配层统一接口”在日报中没有明确发生，容易被问细节。

建议换成以下真实案例：

案例 1：微信小游戏打包失败与资源加载问题

- Situation：微信小游戏打包后出现插件冲突、首包加载、DLL 下载失败、缓存未更新等问题。
- Task：保证项目能在手机端跑通。
- Action：排查命名空间冲突、关闭 ILRuntime Debug、调整首包资源加载、清理服务器缓存和端口、检查 Hotfix DLL。
- Result：完成微信小游戏打包和手机真机调试。
- Learning：打包问题要从插件、资源、缓存、配置、脚本挂载多个维度排查。

案例 2：UI 点击穿透与按钮无响应

- Situation：秘籍、宝箱、属性详情等界面出现按钮点不到或点击穿透。
- Task：保证局内外 UI 交互稳定。
- Action：检查 Canvas、Raycaster、Image、Button、渲染层级和透明点击区域。
- Result：修复多个界面交互问题，并总结 UI 预制体规范。
- Learning：UI 问题常常不是逻辑错误，而是层级和射线检测规则不统一。

案例 3：ILRuntime/异步时序问题

- Situation：await、TryWait、out 参数、局部对象跨 await 后出现异常。
- Task：避免热更环境下逻辑错乱或游戏退出卡死。
- Action：减少 out 跨 await 传参，改用 return；TryWait 后判断返回值；关键异步逻辑集中顺序执行。
- Result：修复小李飞刀传参、关闭游戏卡死、技能时序等问题。
- Learning：热更环境中异步写法要比普通 C# 更谨慎。

### 6. “团队协作”部分要更贴近实际协作对象

原文写“与策划”“与后端联调 Mock 数据”，日报里更明显的协作对象是小恒、果哥，以及热更层/API/表格/资源相关沟通。

建议改成：

- 与同事沟通 API 缺失，例如实体子控件获取、UI API、热更层源码阅读。
- 与同事协作处理性能问题，例如骨骼动画转帧动画、碰撞改距离检测、资源服务器动态加载。
- 与表格/资源相关人员协作，例如数据表错误、图标路径、字段占位符、颜色标记处理。

不要重点写“后端 Mock 数据”，除非实际做过。

### 7. “学习能力体系化”方向可行，但要避免口号化

原文中 “Read the Fking Source Code” 不适合正式实习汇报，建议删除或换成正式表达。

建议改成：

- 读项目已有代码：先模仿现有写法，再尝试扩展和优化。
- 最小闭环验证：用 Demo 或小功能先验证，再移植到项目。
- 问题沉淀：把 UI、打包、异步、配置表、性能优化问题整理成经验清单。

日报里比较能支撑的内容包括：

- 阅读俄罗斯方块、方块跳动、类“元气骑士”、是功夫相关代码。
- 通过 Demo 验证热更框架和 API。
- 多次总结异步、UI、打包、资源和性能问题。

### 8. “未来规划”需要改成 Unity/游戏客户端方向

原文写“深入钻研 Cocos/Unity 源码机制”，但你的实习主线是 Unity/C#/ILRuntime/小游戏/移动端，不建议突然加入 Cocos。

建议改成：

短期：

- 深入 Unity 客户端开发流程。
- 加强 C# 异步、热更、资源加载、UI 框架、配置表工具链理解。
- 熟悉移动端性能分析和打包发布流程。

中期：

- 能独立负责一个完整客户端模块，从需求拆解、实现、联调、测试到打包。
- 建立稳定的 Bug 排查和性能优化方法。

长期：

- 成为具备玩法开发、工程化能力和性能意识的游戏客户端程序。

## 三、建议改成的 PPT 流程

如果汇报时间在 15-20 分钟左右，建议控制在 14-18 页，不建议做 36 页。当前文档写的页数和时间合计接近 35 分钟，除非公司明确给这么长时间，否则容易讲不完。

推荐结构如下：

### Part 1：实习概况与工作主线，2 页

第 1 页：个人与实习信息

- 岗位：游戏客户端程序实习生
- 时间：2026-02-27 至 2026-06-30
- 技术栈：Unity、C#、ILRuntime、UniTask、微信小游戏、安卓、iOS

第 2 页：三个月时间线

- 2 月底至 3 月初：环境熟悉、C#、Unity、热更学习
- 3 月：Demo 移植与热更框架验证
- 4 月：类“是功夫”战斗、技能、UI、打包测试
- 6 月：局外系统、秘籍、勾玉、阵法、兵器、淬炼、打包收尾

### Part 2：主要工作成果，5-6 页

建议按模块讲，不要按日期流水账。

1. 热更框架与 Demo 移植
2. 战斗系统与角色/敌人逻辑
3. 技能、武器与掉落系统
4. UI 与局外养成系统
5. 配置表与资源接入
6. 多端打包与真机测试

每页结构建议统一：

- 左侧：截图或流程图
- 右侧：我负责什么、解决了什么、产出是什么

### Part 3：重点案例复盘，4-5 页

选择 3 个最能体现程序能力的案例即可。

推荐案例：

1. UI 点击与层级问题复盘
2. ILRuntime/异步时序问题复盘
3. 微信小游戏/安卓/iOS 打包问题复盘
4. 移动端性能优化复盘

每个案例使用：

问题背景 -> 排查过程 -> 解决方案 -> 结果 -> 收获

### Part 4：能力成长，2-3 页

建议分成三类：

- 技术能力：Unity、C#、热更、UI、配置表、打包、性能。
- 工程能力：模块拆分、通用 UI、enum 管理、资源规范、问题清单。
- 问题解决能力：从“看报错”到“定位根因”，从“改 Bug”到“总结规则”。

### Part 5：不足与规划，1-2 页

不足建议写真实但不自贬：

- 对底层渲染、资源加载和平台差异理解还不够深。
- 对复杂异步和热更环境问题还需要更多项目经验。
- 性能优化目前更多是经验总结，后续需要结合工具和数据分析。

规划：

- 深入 Unity 客户端工程体系。
- 加强移动端性能分析、资源管理和自动化打包流程。
- 提升独立负责完整模块的能力。

### Part 6：总结致谢，1 页

建议总结句：

“这三个月让我从单纯实现功能，逐步转向关注模块稳定性、可维护性、跨平台适配和团队协作，初步建立了游戏客户端开发的工程化思维。”

## 六、最推荐的汇报主线

建议整份 PPT 围绕一句话展开：

“我从一个熟悉 UE/C++ 的实习生，逐步适应 Unity/C#/热更开发，并在 Demo 移植、项目功能开发、多端打包和性能优化中，建立了游戏客户端程序的工程化思维。”

这条主线和日报内容最匹配，也能自然解释你的成长：

- 为什么一开始在学 C#、Unity、ILRuntime。
- 为什么中间做 Demo 移植和框架验证。
- 为什么后面能参与“是功夫”的多个系统。
- 为什么汇报重点不是“我写了很多功能”，而是“我开始理解游戏客户端开发的完整链路”。
