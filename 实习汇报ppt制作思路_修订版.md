## Part 2：工作方法与工业级开发思维（重点，约 10 页，10 分钟）

### 核心

这一部分是汇报重点。建议保留原来“之前写法 vs 更成熟写法”的对比形式，但把概念改得更贴合日报。

不建议继续使用“反脆弱设计”这种偏大的词，可以改成：

**从能用就行，到稳定可维护。**

### 要点 1：从“能跑通”到“考虑异常和边界”

#### 场景

在配置表、资源加载、UI 显示、热更 DLL、打包流程中，经常会遇到字段不一致、资源路径错误、表格没更新、平台 API 不支持、按钮点不到等问题。

#### 之前容易出现的思路

只关注正常流程：

- 表格能读出来就继续写逻辑。
- UI 能显示就默认没有问题。
- PC 能跑通就默认手机端也能跑。
- 一个功能调通后就继续写下一个功能。

#### 实习后形成的方法

开始主动考虑异常和边界：

- 配置表字段名、字段类型和代码字段需要严格对应。
- 图片、预制体、技能图标等资源路径需要和表格规则统一。
- UI 按钮是否可点击，不只看 Button，还要检查 Image、Raycaster、Canvas 和层级。
- 打包前要检查服务器、缓存、热更 DLL、场景配置、脚本挂载和平台 API。
- await、TryWait、out 参数等写法在热更环境中要更谨慎。

#### PPT 展示方式

画一张“功能正常流程 + 异常分支”的流程图。

例如：配置表读取流程

读取配置表 -> 字段名检查 -> 类型检查 -> 资源路径检查 -> 默认值/兜底处理 -> 功能逻辑执行

#### 体会

“客户端开发不只是把正常流程写通，还要提前考虑资源、配置、平台和运行时环境可能出错的情况。越靠后的问题，排查成本越高。”

### 要点 2：从“直接写功能”到“复用和规范”

#### 场景

在练功房、属性详情、秘籍、兵器、奖励弹窗等 UI 开发中，多个界面都有类似的弹窗、图标框、详情面板、空白点击关闭等交互。

#### 原始写法

每个界面单独做：

- 单独写按钮点击。
- 单独写弹窗打开关闭。
- 单独处理图标显示。
- 单独处理层级和射线检测。

这种写法短期能完成，但后续容易出现重复代码和不一致的交互问题。

#### 实习后形成的方法

开始考虑复用和统一规则：

- 制作通用弹窗 UI。
- 复用技能/武器图标展示框。
- 将空白点击关闭、详情面板、返回按钮等交互方式统一。
- 对 Canvas、Raycaster、Image、Button 的使用形成固定检查规则。
- 对技能、勾玉等系统尽量使用 enum 管理，减少字符串硬编码。

#### 注意

不要在 PPT 中写“已经搭建完整组件库”或“开发效率提升 30%”，除非有明确数据。可以改成：

“通过通用预制体和统一交互规则，减少了后续相似界面的重复制作和层级问题。”

#### 体会

“功能做多以后，真正影响效率的不是单个按钮怎么写，而是同类问题能不能被统一处理。”

### 要点 3：从“遇到 Bug 就改”到“建立排查路径”

#### 场景

实习中遇到过很多类型的问题：

- UI 点不到或点击穿透。
- 微信小游戏打包失败或资源不更新。
- iOS 平台 API 不兼容。
- ILRuntime Debug/Release 行为差异。
- await 后对象或 out 参数异常。
- 敌人数量多时性能下降。
- meta 文件损坏导致资源引用异常。

#### 原始思路

看到报错后直接搜索报错，或者尝试改一两个可疑点。

#### 实习后形成的方法

先判断问题类型，再走排查路径：

- UI 问题：先查 Canvas、Raycaster、Image、Button、层级、透明区域。
- 配置表问题：先查字段名、字段类型、路径、表格是否更新到热更层。
- 打包问题：先定位 failure/fail 关键错误，再查插件、平台 API、场景、资源、热更 DLL。
- 异步问题：先查 await 前后对象生命周期、TryWait 返回值、Forget 调用顺序。
- 性能问题：先看 GC、碰撞、动画、资源大小、Shader、UI 叠层和敌人数量。

#### PPT 展示方式

用一页“问题分类表”展示：

问题类型 | 常见现象 | 排查入口 | 最终沉淀

#### 体会

“解决问题不能只靠经验猜，应该逐步建立自己的排查清单。清单越完整，越能减少重复踩坑。”

### 要点 4：从“PC 能跑”到“跨平台意识”

#### 场景

项目不仅要在编辑器和 PC 上跑，还要打包到微信小游戏、安卓、iOS 等平台。

#### 真实问题

- 微信小游戏打包插件与 ILRuntime 引用冲突。
- 首包资源加载方式不正确。
- 本地服务器和缓存导致打包后不是最新资源。
- 安卓打包出现 PlayerAndEditor、DLL 未更新、Json 文件损坏等问题。
- iOS 不支持部分 API，需要做平台限制。
- 包体过大、资源动态加载、图片压缩、Shader 和动画优化都影响移动端表现。

#### 方法沉淀

打包前建立检查项：

- 热更 DLL 是否重新生成。
- 本地服务器是否关闭或地址是否正确。
- 微信开发者工具缓存是否清理。
- 场景是否加入 Build Settings。
- 平台不支持 API 是否隔离。
- 非 Editor 目录是否误引用 UnityEditor。
- 资源路径、meta 文件、GUID 是否正常。

#### 体会

“客户端功能在编辑器里能跑，只代表完成了第一步。真正接近项目交付时，还要面对平台、资源、缓存、包体和性能问题。”

## Part 3：踩坑实录与解决策略（约 5 页，5 分钟）

### 目的

这一部分非常适合保留。不要把踩坑讲成“我犯了很多错”，而是讲成“我如何从问题中提炼方法”。

建议使用 STAR-L：

- Situation：背景
- Task：目标
- Action：行动
- Result：结果
- Learning：收获

### 案例 1：微信小游戏打包和真机调试

#### Situation

类“元气骑士”Demo 和后续项目需要打包到微信小游戏和手机端测试，但过程中遇到插件冲突、资源加载、缓存、DLL 下载失败等问题。

#### Task

让项目在手机端成功运行，并保证热更内容和资源能正确加载。

#### Action

- 处理 ILRuntime 和微信小游戏打包插件的命名空间冲突。
- 关闭 ILRuntime Debug 相关功能，避免 Player/Editor 冲突。
- 调整首包资源加载方式。
- 检查 Hotfix DLL 是否正确生成和下载。
- 清理本地服务器、端口和微信开发者工具缓存。
- 通过 IP、USB 网络共享、HTTP 权限等方式排查真机连接问题。

#### Result

完成微信小游戏打包和手机真机调试，积累了移动端打包排查流程。

#### Learning

打包问题往往不是单个代码问题，而是插件、资源、缓存、服务器、平台配置共同作用的结果。

### 案例 2：UI 点击穿透和按钮无响应

#### Situation

秘籍界面、宝箱、属性详情、兵器详情等界面中出现过按钮点不到、点击穿透、关闭不生效、图片被挡住等问题。

#### Task

保证局内外 UI 交互稳定，点击符合预期。

#### Action

- 检查按钮是否有 Image 或可射线检测区域。
- 检查 Raycaster 是否挂在正确层级。
- 检查 Canvas 是否造成层级混乱。
- 使用透明 Image + Button 实现空白区域关闭弹窗。
- 调整渲染层级，解决勾玉贴图遮挡宝箱的问题。
- 对复用 UI 预制体的根节点 Canvas/Raycaster 做统一处理。

#### Result

修复多个 UI 点击问题，并形成了 UI 交互排查规则。

#### Learning

UI 问题很多时候不是代码逻辑错，而是层级、射线检测、渲染顺序和预制体结构不统一。

### 案例 3：热更环境下的异步时序问题

#### Situation

在 ILRuntime/热更环境中，遇到 await 后对象引用异常、out 参数传递失败、TryWait 提前退出导致卡死等问题。

#### Task

保证技能、武器、UI 和退出流程在异步逻辑下稳定执行。

#### Action

- 对 TryWait 返回值做判断，失败时及时 return。
- 减少 out 参数跨 await 传值，改用 return 对象或数组。
- 将关键异步流程集中管理，避免多个 Forget 同时触发导致顺序混乱。
- 对 await 前后的对象生命周期进行额外检查。

#### Result

修复小李飞刀传参、关闭游戏卡死、技能时序等问题。

#### Learning

热更环境中的异步写法不能完全照搬普通 C# 经验，必须更重视生命周期和执行顺序。

### 案例 4：性能与资源问题

#### Situation

敌人数量增加、技能特效增多、图片资源过大后，出现帧率下降、图片消失、资源加载不稳定等问题。

#### Task

减少移动端压力，提升运行稳定性。

#### Action

- 减少循环中的 GC 开销。
- 将部分碰撞检测改为距离检测。
- 将骨骼动画转换为帧动画。
- 压缩过大的秘籍图标，考虑使用 SpriteAtlas。
- 控制 Shader、贴图尺寸、UI 叠层、粒子数量和资源加载方式。
- 对敌人预制体预热、实体注册、视觉状态批处理等流程进行总结。

#### Result

对移动端性能问题形成了更完整的认识，并能从资源、动画、碰撞、UI、渲染等多个角度排查。

#### Learning

性能优化不是最后才做的事情，很多问题从资源规范、逻辑写法和系统设计阶段就已经开始影响结果。

## Part 4：团队协作与非技术软技能（约 3 页，3 分钟）

### 目的

保留原来“非技术软技能”的部分，但案例建议从“后端 Mock”和“策划沟通”改成更符合日报的协作内容。

### 要点 1：从单兵写功能到主动对齐接口和规则

#### 真实场景

在开发过程中，经常需要和同事确认：

- 热更层 API 能否支持某个功能。
- 是否能获取预制体子节点。
- UI 相关接口是否需要补充。
- 表格字段、资源路径、图标命名是否一致。
- 性能优化方案是否需要改动画、碰撞或资源加载方式。

#### 体会

“客户端开发不是自己把逻辑写完就结束。很多问题需要和框架、资源、表格、打包流程对齐，提前沟通能减少后期返工。”

### 要点 2：描述问题时从“它坏了”到“给出可复现路径”

#### 方法

遇到问题时，尽量说明：

- 出现在哪个平台。
- 触发步骤是什么。
- 当前表现是什么。
- 期望表现是什么。
- 已经排查过哪些方向。
- 是否有报错、截图或日志。

#### 体会

“问题描述越清晰，协作效率越高。尤其是打包、资源和 UI 层级问题，如果没有复现路径，很容易陷入猜测。”

### 要点 3：理解需求背后的目标

#### 场景

技能、武器、掉落、UI 表现、伤害飘字、宝箱、阵法等功能，本质上不只是实现逻辑，还要服务玩家反馈和游戏体验。

#### 体会

“游戏客户端程序不能只看代码是否正确，还要关注表现是否清晰、反馈是否及时、操作是否符合玩家直觉。”

## Part 5：学习能力的体系化构建（重点，约 5 页，8 分钟）

### 目的

保留原来“学习内容不是重点，学习方法才是重点”的方向。把粗口表达和泛化内容改成正式、真实的版本。

### 方法 1：先读已有代码，再扩展功能

#### 场景

刚接触 Unity、C#、ILRuntime 和热更项目时，直接写新功能效率不高。

#### 做法

- 先阅读已有 Demo 和项目代码。
- 通过方块跳动、俄罗斯方块、类“元气骑士”等 Demo 理解代码分层和 API 调用方式。
- 先模仿项目已有写法，确认能跑通后，再尝试扩展和优化。

#### 体会

“进入已有项目时，先理解项目规则比直接创新更重要。遵守已有结构，才能让自己的代码更容易接入团队工程。”

### 方法 2：用最小闭环验证不确定点

#### 场景

面对新功能、新资源、新打包方式时，直接放进大系统里调试成本很高。

#### 做法

- 先用 Demo 或小功能验证核心逻辑。
- 例如先通过 Demo 验证热更框架，再移植到实际项目。
- 对技能、UI、资源和打包问题，先确认最小路径是否成立，再扩大到完整功能。

#### 体会

“最小闭环可以降低不确定性。先证明一个小功能能跑，再接入完整系统，排查会更清晰。”

### 方法 3：问题沉淀成清单

#### 场景

实习中很多问题会重复出现，例如 UI 点击、配置表读错、资源路径错误、打包缓存、异步时序等。

#### 做法

把问题沉淀成检查清单：

- UI 检查清单：Canvas、Raycaster、Image、Button、层级、透明点击区域。
- 打包检查清单：DLL、服务器、缓存、场景、平台 API、UnityEditor 引用、meta 文件。
- 配置表检查清单：字段名、字段类型、路径、资源命名、表格是否更新。
- 异步检查清单：await 前后对象生命周期、TryWait 返回值、Forget 调用顺序。

#### 体会

“记录问题不是为了证明自己踩过坑，而是为了下次更快定位，并减少团队里其他人重复踩坑。”

### 方法 4：从结果倒推能力短板

#### 场景

当打包、性能、资源、异步问题暴露出来时，能看到自己在底层机制上的理解还不够。

#### 做法

- 从实际问题倒推需要补的知识。
- 打包问题倒推平台差异、资源加载、热更 DLL 流程。
- 性能问题倒推 GC、动画、碰撞、DrawCall、Shader、贴图压缩。
- 异步问题倒推 UniTask、await、生命周期和热更环境差异。

#### 体会

“项目中的问题会告诉我最应该补什么，而不是漫无目的地学所有内容。”

## Part 6：不足与未来规划（约 2 页，3 分钟）

### 不足

建议保持谦逊，但不要写得太虚。

可以写：

1. 对 Unity 底层渲染、资源加载和平台差异理解还不够深入。
2. 对复杂异步流程和热更环境下的生命周期问题，还需要更多项目经验。
3. 性能优化目前更多是从问题中总结经验，后续需要更系统地结合工具和数据分析。
4. 对完整商业项目的上线流程、自动化打包和长期维护经验还不足。

### 未来规划

#### 短期
