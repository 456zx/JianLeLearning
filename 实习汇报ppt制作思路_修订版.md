# 实习汇报 PPT 制作思路

## 汇报主线（共37分钟左右）

**从完成具体功能，到具备项目分析、独立学习、问题排查和持续成长的游戏客户端开发能力。**

这次汇报不建议只按“我做了哪些功能”平铺直叙，而是按成长路径展开：

- 我在实习项目中参与了哪些真实开发工作。
- 我如何利用其余时间自学，并产出可展示的 Demo 和知识沉淀。
- 我是否开始理解一个游戏项目在开发前应该如何分析和拆解。
- 我如何从问题中形成更成熟的开发方法和排查路径。
- 我后续准备怎样把学习能力变成稳定、可持续的能力体系。

整体建议调整为 6 个 Part：

1. 工作内容展示
2. 自学内容展示
3. 项目总体框架分析与游戏开发流程
4. 工作方法、工业级开发思维与踩坑解决策略
5. 学习能力的体系化构建
6. 自我评价与未来规划

## Part 1：工作内容展示（约 5-6 页，7分钟）

### 目的

这一部分用于快速说明“我在公司参与过什么、接触过哪些模块”。不要在这里展开所有技术细节，而是作为后面方法论、项目分析和学习能力展示的铺垫。

### 展示重点

可以按实习阶段展示：

1. 适应期：熟悉 Unity、C#、ILRuntime、UniTask、热更框架和项目开发流程。
2. 验证期：参与俄罗斯方块、类“元气骑士”等 Demo 的阅读、移植和框架验证。
3. 项目期：参与类“是功夫”项目的客户端功能开发，包括战斗、技能、武器、UI、配置表、资源接入等。
4. 测试与优化期：参与微信小游戏、安卓、iOS 打包测试，处理平台适配、资源加载、性能和工程配置问题。

### PPT 写法

建议采用“截图 + 关键词 + 一句话说明”，避免大段文字。

可安排页面：

1. 实习整体时间线：2 月底到 6 月底。
2. 技术栈总览：Unity、C#、ILRuntime、UniTask、Json/TE 表格、微信小游戏、安卓、iOS。
3. Demo 验证：类“元气骑士”移植、热更框架和 API 验证。
4. 战斗系统：玩家、敌人、攻击、伤害、血条、掉落。
5. 技能与武器：小李飞刀、金钟罩、火云掌、武器飞行和命中逻辑等。
6. UI 与局外系统：练功房、秘籍、淘秘籍、勾玉、阵法、兵器、淬炼。
7. 配置表与资源：敌人、技能、勾玉、阵法、兵器、词条等配置驱动。
8. 打包与测试：微信小游戏、安卓、iOS 真机和平台问题处理。

### 建议话术

“这些内容看起来比较多，但我这次汇报不会重点讲每个功能怎么写。对我来说更重要的是，在这些功能和问题背后，我开始理解游戏客户端开发不是把功能写出来就结束，而是要考虑稳定性、可维护性、资源、平台、性能和开发流程。”

## Part 2：自学内容展示（约 4-5页，5分钟）

### 目的

这一部分新增出来，用来体现实习工作之外的自学能力和持续产出能力。重点不是罗列“我学了哪些知识”，而是展示：我能主动发现短板、安排学习内容，并把学习结果转化为 Demo、笔记和可复用经验。

### 展示主线

建议用一句话概括：

**从跟随公司 Demo 学习，到独立在家制作卡牌游戏 Demo，再到补齐 C#、Git、Linux 等基础能力。**

### 展示内容 1：公司内第一个类“元气骑士”项目

#### 展示重点

- 作为刚进入公司后接触的第一个完整游戏 Demo，用来熟悉 Unity 项目结构和热更开发方式。
- 重点展示角色移动、攻击、敌人、技能、掉落、UI、资源加载等基础模块。
- 说明这个 Demo 对自己的意义：不是简单照着做，而是通过它理解“游戏最小闭环”和项目分层。

#### PPT 页面建议

1. Demo 截图或运行画面。
2. 核心模块拆解：角色、敌人、技能、掉落、UI、配置。
3. 学到的内容：Unity 生命周期、热更层调用、资源路径、配置驱动、移动端打包意识。

### 展示内容 2：在家自学制作的卡牌游戏 Demo

#### 展示重点

这个 Demo 可以作为“独立学习能力”的重点展示，因为它体现了你不只是在公司任务中学习，也能在家主动做一个新的游戏类型。

可以突出：

- 自主确定玩法方向：卡牌游戏 Demo。
- 自主拆解核心系统：牌库、手牌、弃牌堆、出牌、回合、敌人、数值结算。
- 自主实现基础 UI 和交互反馈。
- 尝试用配置或数据结构管理卡牌效果，避免所有逻辑硬编码。

#### PPT 页面建议

1. 卡牌 Demo 整体截图：主界面、战斗界面或出牌界面。
2. 核心循环：抽牌 -> 出牌 -> 结算效果 -> 敌方行动 -> 回合切换。
3. 系统拆解：卡牌数据、牌堆管理、回合管理、战斗结算、UI 刷新。
4. 对比公司项目的收获：同样是游戏客户端，但卡牌游戏更强调数据结构、状态流转和 UI 反馈。

### 展示内容 3：基础知识自学沉淀

结合文件夹里的学习材料，可以分成几类展示：

#### C# 与 Unity 相关

- 数据类型、基本语法、委托和强制转换。
- 协程、异步、UniTask。
- Shader 基础尝试。

建议说明：这些内容不是孤立学习，而是为理解项目中的异步流程、热更逻辑、技能实现和资源表现服务。

#### Git 与版本管理

- 理解提交、分支、合并、冲突处理和版本回退。
- 认识到版本管理不是“会提交代码”这么简单，而是团队协作、问题追踪和风险控制的一部分。

如果暂时没有单独的 Git 截图，可以放一页“版本管理学习清单”，后续再补命令行截图或笔记截图。

#### Linux 与工程环境

- 学习文件属性、权限、路径、常用命令等基础内容。
- 结合打包、服务器、本地路径和资源加载问题，说明 Linux/命令行基础对工程排查有帮助。

### 建议话术

“我把自学内容分成两类：一类是能跑起来的 Demo，证明我能把知识落到实际项目里；另一类是基础知识补齐，比如 C#、Git、Linux。这些内容短期看可能不如具体功能显眼，但它们决定了后续遇到复杂问题时，我能不能自己继续往下查、往下学。”

## Part 3：项目总体框架分析与游戏开发流程（约 6-7页，8分钟）

### 目的

这一部分由原来的“团队协作与非技术软技能”调整而来，不再重点讲沟通能力，而是展示你对项目开发前准备工作的理解。它的作用是证明：你不只是会写功能，也开始理解一个游戏项目在开发前应该如何分析、拆解和组织。

### 展示重点 1：游戏最小循环分析

#### 核心概念

游戏开发前要先分析“最小可玩闭环”，也就是玩家最基本的一轮体验能否成立。

可以用类“是功夫”或卡牌 Demo 举例：

- 类动作/割草项目：进入战斗 -> 移动 -> 攻击敌人 -> 造成伤害 -> 获得掉落/成长 -> 继续战斗。
- 卡牌 Demo：进入战斗 -> 抽牌 -> 出牌 -> 结算效果 -> 敌方行动 -> 回合切换 -> 战斗结束。

#### PPT 展示方式

画两张闭环图：

1. 公司项目的战斗最小循环。
2. 自学卡牌 Demo 的回合制最小循环。

这样可以自然连接 Part 1 的工作内容和 Part 2 的自学内容。

### 展示重点 2：游戏开发流程

建议按从前到后的流程展示：

1. 需求理解：明确玩法目标、操作方式、核心反馈和完成标准。
2. 最小闭环验证：先让最核心的玩法跑起来。
3. 系统拆分：拆成角色、敌人、技能、武器、UI、配置、资源、音效、打包等模块。
4. 数据和资源准备：确定配置表字段、资源路径、命名规范、预制体结构。
5. 功能开发：按模块逐步实现，并保持和已有框架一致。
6. 联调测试：检查 UI、配置、资源、平台、性能和异常情况。
7. 打包验证：微信小游戏、安卓、iOS 等平台分别测试。
8. 问题沉淀：把重复问题整理成检查清单和经验记录。

### 展示重点 3：热更层开发文件职责分类

这部分可以体现你对项目结构的理解。建议不要展示太多具体文件名，而是按职责分类：

| 类别 | 主要职责 | PPT 展示重点 |
| --- | --- | --- |
| 入口与流程控制 | 初始化、模块加载、场景切换、主流程调度 | 项目如何启动和进入玩法 |
| 战斗逻辑 | 玩家、敌人、技能、武器、伤害、掉落 | 核心玩法如何运行 |
| UI 层 | 界面打开关闭、按钮事件、弹窗、详情面板 | 玩家操作和信息反馈 |
| 配置读取 | 表格解析、字段映射、默认值、数据查询 | 功能如何由数据驱动 |
| 资源管理 | 图标、预制体、特效、音效加载 | 资源路径和加载规则 |
| 平台与打包适配 | 微信小游戏、安卓、iOS 差异处理 | 项目如何真正跑到设备上 |
| 工具与通用方法 | 通用弹窗、枚举、扩展方法、检查工具 | 减少重复开发和维护成本 |

### 建议话术

“以前我更关注一个功能具体怎么写。实习后我发现，功能开发前的分析同样重要：先确认最小循环，再拆系统和文件职责，最后再进入具体实现。这样做可以减少后面配置、资源、UI 和打包阶段的返工。”

## Part 4：开发思维与踩坑解决（约 7-8 页，8 分钟）

### 目的

这一部分由原来的“工作方法与工业级开发思维”和“踩坑实录与解决策略”合并而来。建议作为整场汇报的重点：用真实问题说明自己从“能写功能”逐步转向“能稳定交付、能定位问题、能沉淀方法”。

### 核心表达

**从能用就行，到稳定可维护；从遇到问题就改，到建立排查路径。**

### 要点 1：从“能跑通”到“考虑异常和边界”

#### 场景

在配置表、资源加载、UI 显示、热更 DLL、打包流程中，经常会遇到字段不一致、资源路径错误、表格没更新、平台 API 不支持、按钮点不到等问题。

#### 方法沉淀

- 配置表字段名、字段类型和代码字段需要严格对应。
- 图片、预制体、技能图标等资源路径需要和表格规则统一。
- UI 按钮是否可点击，不只看 Button，还要检查 Image、Raycaster、Canvas 和层级。
- 打包前要检查服务器、缓存、热更 DLL、场景配置、脚本挂载和平台 API。
- await、TryWait、out 参数等写法在热更环境中要更谨慎。

#### 体会

“客户端开发不只是把正常流程写通，还要提前考虑资源、配置、平台和运行时环境可能出错的情况。越靠后的问题，排查成本越高。”

### 要点 2：从“直接写功能”到“复用和规范”

#### 场景

在练功房、属性详情、秘籍、兵器、奖励弹窗等 UI 开发中，多个界面都有类似的弹窗、图标框、详情面板、空白点击关闭等交互。

#### 方法沉淀

- 制作通用弹窗 UI。
- 复用技能/武器图标展示框。
- 将空白点击关闭、详情面板、返回按钮等交互方式统一。
- 对 Canvas、Raycaster、Image、Button 的使用形成固定检查规则。
- 对技能、勾玉等系统尽量使用 enum 管理，减少字符串硬编码。

#### 体会

“功能做多以后，真正影响效率的不是单个按钮怎么写，而是同类问题能不能被统一处理。”

### 要点 3：从“遇到 Bug 就改”到“建立排查路径”

#### 问题分类表

| 问题类型 | 常见现象 | 排查入口 | 最终问题分类 |
| --- | --- | --- | --- |
| UI 问题 | 点不到、点击穿透、关闭无效 | Canvas、Raycaster、Image、Button、层级 | UI 交互检查清单 |
| 配置表问题 | 字段为空、资源找不到、数据不生效 | 字段名、字段类型、路径、表格更新 | 配置表检查清单 |
| 打包问题 | 微信小游戏/安卓/iOS 运行异常 | failure/fail 日志、插件、平台 API、DLL、缓存 | 打包前检查清单 |
| 异步问题 | await 后对象异常、流程卡死 | 生命周期、TryWait 返回值、Forget 调用顺序 | 异步流程检查清单 |
| 性能问题 | 帧率下降、资源加载慢 | GC、碰撞、动画、Shader、贴图、UI 叠层 | 移动端性能排查清单 |

#### 体会

“解决问题不能只靠经验猜，应该逐步建立自己的排查清单。清单越完整，越能减少重复踩坑。”

### 真实案例 1：微信小游戏打包和真机调试

#### Situation

类“元气骑士”Demo 和后续项目需要打包到微信小游戏和手机端测试，但过程中遇到插件冲突、资源加载、缓存、DLL 下载失败等问题。

#### Action

- 处理 ILRuntime 和微信小游戏打包插件的命名空间冲突。
- 关闭 ILRuntime Debug 相关功能，避免 Player/Editor 冲突。
- 调整首包资源加载方式。
- 检查 Hotfix DLL 是否正确生成和下载。
- 清理本地服务器、端口和微信开发者工具缓存。
- 通过 IP、USB 网络共享、HTTP 权限等方式排查真机连接问题。

#### Learning

打包问题往往不是单个代码问题，而是插件、资源、缓存、服务器、平台配置共同作用的结果。

### 真实案例 2：UI 点击穿透和按钮无响应

#### Situation

秘籍界面、宝箱、属性详情、兵器详情等界面中出现过按钮点不到、点击穿透、关闭不生效、图片被挡住等问题。

#### Action

- 检查按钮是否有 Image 或可射线检测区域。
- 检查 Raycaster 是否挂在正确层级。
- 检查 Canvas 是否造成层级混乱。
- 使用透明 Image + Button 实现空白区域关闭弹窗。
- 调整渲染层级，解决勾玉贴图遮挡宝箱的问题。
- 对复用 UI 预制体的根节点 Canvas/Raycaster 做统一处理。

#### Learning

UI 问题很多时候不是代码逻辑错，而是层级、射线检测、渲染顺序和预制体结构不统一。

### 真实案例 3：热更环境下的异步时序问题

#### Situation

在 ILRuntime/热更环境中，遇到 await 后对象引用异常、out 参数传递失败、TryWait 提前退出导致卡死等问题。

#### Action

- 对 TryWait 返回值做判断，失败时及时 return。
- 减少 out 参数跨 await 传值，改用 return 对象或数组。
- 将关键异步流程集中管理，避免多个 Forget 同时触发导致顺序混乱。
- 对 await 前后的对象生命周期进行额外检查。

#### Learning

热更环境中的异步写法不能完全照搬普通 C# 经验，必须更重视生命周期和执行顺序。

### 真实案例 4：性能与资源问题

#### Situation

敌人数量增加、技能特效增多、图片资源过大后，出现帧率下降、图片消失、资源加载不稳定等问题。

#### Action

- 减少循环中的 GC 开销。
- 将部分碰撞检测改为距离检测。
- 将骨骼动画转换为帧动画。
- 压缩过大的秘籍图标，考虑使用 SpriteAtlas。
- 控制 Shader、贴图尺寸、UI 叠层、粒子数量和资源加载方式。
- 对敌人预制体预热、实体注册、视觉状态批处理等流程进行总结。

#### Learning

性能优化不是最后才做的事情，很多问题从资源规范、逻辑写法和系统设计阶段就已经开始影响结果。

## Part 5：学习能力的体系化构建（约 5 页，6 分钟）

### 目的

这一部分保留原来“学习内容不是重点，学习方法才是重点”的方向。Part 2 已经展示了自学成果，这里要进一步说明：这些学习不是零散的，而是逐步形成了一套适合自己的学习体系。

### 方法 1：先读已有代码，再扩展功能

#### 场景

刚接触 Unity、C#、ILRuntime 和热更项目时，直接写新功能效率不高。

#### 做法

- 先阅读已有 Demo 和项目代码。
- 通过方块跳动、俄罗斯方块、类“元气骑士”等 Demo 理解代码分层和 API 调用方式。
- 先模仿项目已有写法，确认能跑通后，再尝试扩展和优化。

#### 体会

“进入已有项目时，先理解项目规则比直接创新更重要。遵守已有结构，才能让自己的代码更容易接入团队工程。”

### 方法 2：用最小闭环验证不确定点

#### 场景

面对新功能、新资源、新打包方式时，直接放进大系统里调试成本很高。

#### 做法

- 先用 Demo 或小功能验证核心逻辑。
- 例如先通过 Demo 验证热更框架，再移植到实际项目。
- 对技能、UI、资源和打包问题，先确认最小路径是否成立，再扩大到完整功能。

#### 体会

“最小闭环可以降低不确定性。先证明一个小功能能跑，再接入完整系统，排查会更清晰。”

### 方法 3：问题沉淀成清单

#### 场景

实习中很多问题会重复出现，例如 UI 点击、配置表读错、资源路径错误、打包缓存、异步时序等。

#### 做法

把问题沉淀成检查清单：

- UI 检查清单：Canvas、Raycaster、Image、Button、层级、透明点击区域。
- 打包检查清单：DLL、服务器、缓存、场景、平台 API、UnityEditor 引用、meta 文件。
- 配置表检查清单：字段名、字段类型、路径、资源命名、表格是否更新。
- 异步检查清单：await 前后对象生命周期、TryWait 返回值、Forget 调用顺序。

#### 体会

“记录问题不是为了证明自己踩过坑，而是为了下次更快定位，并减少团队里其他人重复踩坑。”

### 方法 4：从结果倒推能力短板

#### 场景

当打包、性能、资源、异步问题暴露出来时，能看到自己在底层机制上的理解还不够。

#### 做法

- 从实际问题倒推需要补的知识。
- 打包问题倒推平台差异、资源加载、热更 DLL 流程。
- 性能问题倒推 GC、动画、碰撞、DrawCall、Shader、贴图压缩。
- 异步问题倒推 UniTask、await、生命周期和热更环境差异。

#### 体会

“项目中的问题会告诉我最应该补什么，而不是漫无目的地学所有内容。”

### 方法 5：把学习成果可视化

#### 做法

- 用 Demo 截图证明学习结果。
- 用流程图证明自己理解了系统关系。
- 用检查清单证明问题有沉淀。
- 用代码笔记证明基础知识有持续补齐。

#### 体会

“学习能力不只体现在学过什么，还体现在能不能把学习过程整理成别人能看懂、自己以后能复用的材料。”
