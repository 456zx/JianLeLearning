## 1. 前置了解

在开始上手热更业务代码之前，需要先建立一个整体认知：热更项目不是“Unity 直接挂脚本驱动全部逻辑”的写法，而是“中间层 API + ILRuntime 热更层 + 热更层”三层协作。

### 1.1 ILRuntime 热更原理

`Kongfu` 示例项目当前采用的是 ILRuntime 方案。它的核心思路是把业务逻辑（热更代码）单独编译成热更 DLL，在运行时再加载并执行。

可以把整个链路理解为下面几步：

1. 开发者在 `HotfixProject~` 中编写热更业务代码。
2. 通过 `HotfixProject~/Hotfix.csproj` 编译出 `HotFix.Kongfu.dll`。
3. 编译产物会被复制到 `Assets/StreamingAssets/`。
4. 游戏运行时，`ILRuntimeHost` 从 `StreamingAssets` 读取 DLL、PDB。

这套方案的优点是：

1. 业务逻辑可以独立于引擎工程迭代。
2. 运行时业务逻辑只加载新的热更程序集，方便更新。

总之：HotfixProject~` 写的是“业务规则”；真正执行它的是运行中的 `ILRuntimeHost`；而 `ILRuntimeHost` 读取的并不是源码，而是 `StreamingAssets` 里的编译产物。

### 1.2 中间层的作用，做了哪些事

本文里的“中间层”，可以理解为 Unity 宿主工程中已经编进包体、负责底层运行能力的部分。它主要位于：

1. `Packages/com.jian.runtime/Runtime`
2. `Packages/com.jian.hotfix.ilruntime/Runtime`

主要负责以下事情：

#### 1. 提供运行时的基本方法

包括但不限于：

1. 游戏生命周期管理

2. 场景和舞台切换

3. UI 创建、显示、关闭、层级管理

4. 实体创建、查找、回收

5. 资源加载

6. 日志、时间、输入、调试能力

7. 数据模块和配置读取

具体方法详见相关api页面文档

这些能力在当前工程里主要收敛到 `ScriptRuntime` 这一层 API 门面中，热更业务只能通过它访问宿主能力。

#### 2. 负责 ILRuntime 加载

它会：

1. 从 `StreamingAssets` 或可选远端 URL 读取热更 DLL/PDB
2. 注册跨域适配器、方法委托、CLR 绑定
3. 反射调用热更入口
4. 创建对象池进行管理

### 1.3 中间层和热更层之间的关系

它们的关系可以概括为：

1. 中间层负责把底层能力整理成热更可调用的稳定接口。
2. 热更层只只通过中间层写业务逻辑，不直接操作底层模块。

### 1.4 `\Kongfu\Assets\StreamingAssets` 下的文件怎么生成的，什么时候生成，有什么用

#### 当前目录里实际能看到的关键产物

当前工程里 `Assets/StreamingAssets/` 下主要有：

1. `HotFix.Kongfu.dll`
2. `HotFix.Kongfu.dll.bytes`
3. `HotFix.dll.bytes`
4. `HotFix.Kongfu.pdb`
5. `HotFix.Kongfu.pdb.bytes`
6. `HotFix.pdb.bytes`
7. `HotFix.Kongfu.deps.json`
8. `resource/*.te`
9. `AssetIndex/` 相关索引文件

#### 这些文件是怎么生成的

它们主要由 `HotfixProject~/Hotfix.csproj` 生成和复制。

当前 `Hotfix.csproj` 中和产物相关的关键配置有：

1. `<OutputPath>..\Assets\StreamingAssets\</OutputPath>`
2. `CopyHotfixAssemblyAsBytes`
3. `SyncExternalTeTables`
4. `CopyHotfixTeTables`

含义分别是：

1. DLL 默认输出到 `Assets/StreamingAssets/`
2. 编译完成后，把 DLL/PDB 复制成 `.bytes` 文件
4. `.te` 表存到 `Assets/StreamingAssets/resource/`

#### 什么时候生成

通常会在你手动编译 `HotfixProject~/Hotfix.csproj` 时.

在当前工程里，最直接的理解就是：

1. 先改 `HotfixProject~` 源码
2. 再编译 `Hotfix.csproj`
3. 产物自动同步到 `Assets/StreamingAssets`

如果你只改了源码，没有重新编译，那么 `StreamingAssets` 里的 DLL 不会更新，运行时就不会生效。

#### 每类文件的作用

##### 1. DLL 格式的文件

`HotFix.Kongfu.dll` 是热更工程真实编译产物，这是 ILRuntime 真正加载的热更程序集。

##### 2. PDB / PDB.bytes

这是调试符号文件

##### 3. `resource/*.te`

这是热更层运行时读取的表文件。

##### 4. `HotFix.Kongfu.deps.json`

这是 .NET 构建生成的依赖描述文件。对于理解业务逻辑不是核心，但它属于热更程序集产物的一部分，通常保留即可。

##### 5. `AssetIndex/`

这部分是资源索引产物，和资源寻址、运行时资源访问有关，不是 ILRuntime 加载 DLL 的核心，但依然属于 `StreamingAssets` 中的运行资源准备部分。

## 2. 快速上手

这一节的目标不是讲所有 API 细节，而是帮助你在最短时间内建立“怎么改、怎么编、怎么跑”的开发闭环。

### 2.1 如何编译和运行此项目

对于 Unity 工程，根目录包含：

1. `Kongfu.sln`
2. Unity 自动生成的 `.csproj`
3. 独立的 `HotfixProject~/Hotfix.csproj`

#### 第一步：打开 Unity 工程

先拿到已经包含了中间层相关内容的项目，打开unity工程

#### 第二步：确认场景和启动环境

工程中主场景需要包含 `ILRuntimeHost` 相关配置。

`ILRuntimeHost` 主要负责：

1. 从 `StreamingAssets` 或远端 URL 读取热更 DLL/PDB
2. 注册跨域适配器和 CLR 绑定
3. 调用热更入口方法
4. 驱动 `Tick` 和 `FixedTick`

![image-20260703163943169](images/hotfix/image-20260703163943169.png)

需要确认场景中包括以下两点：

1. ilruntimehost脚本存在
2. ilruntimehost脚本url已经配置（置空表示默认加载streamingasset下的dll文件）

#### 第三步：创建和编译热更代码

在创建热更工程之前，需要注意以下几点：

1.`HotfixProject~/`不被引擎识别，所以它作为一个单独的工程，拥有.csproject文件

2.在.csproject文件里，定义项目真正的输出目录为`Assets/StreamingAssets/`,也就是编译的产物dll会被输出到这里。

![image-20260703164622817](images/hotfix/image-20260703164622817.png)

3.所以每次更改热更文件夹下的文件，都需要手动编译热更工程。

具体操作的步骤是：

1.手工创建 `HotfixProject~` 目录
这个目录不是 Unity 自动生成的，是你自己建的独立 C# 工程目录。
最少先放：

- `Hotfix.csproj`
- `Entry.cs`

2.配好 `Hotfix.csproj`
最关键的是这几项：

![image-20260703165538760](images/hotfix/image-20260703165538760.png)

- `AssemblyName=HotFix.你的项目名`

- `RootNamespace=HotFix.你的项目名`

- `OutputPath=..\Assets\StreamingAssets\`

- 引用 `Assembly-CSharp.dll`

- 引用 `Jian.Runtime.dll`

- 引用 `UniTask.dll`

- 引用 `UnityEngine.*.dll`

3 手动编译热更代码（编译才会更新dll文件）

  在vs2022中是手动将HotfixProject~\Hotfix.csproj添加进来

  ![image-20260703171942780](images/hotfix/image-20260703171942780.png)

![image-20260703172224716](images/hotfix/image-20260703172224716.png)

后面每次更改热更层文件只需要手动编译hotfix文件夹即可

#### 第五步：运行项目

进入引擎点击 Play Mode 后，会执行下面这条运行链路：

1. `ILRuntimeHost` 开始加载
2. 从 `StreamingAssets` 读取 `HotFix.dll` 
3. 创建 ILRuntime `AppDomain`
4. 初始化 CLR 绑定
5. 调用 `HotFix.Entry.Run()`
6. 后续每帧调用 `HotFix.Entry.Tick()`
7. 后续每个物理帧调用 `HotFix.Entry.FixedTick()`

### 2.2 开发时怎么连接到中间层

热更项目只能通过中间层连接。

#### 通用能力通过 `ScriptRuntime`

常见能力包括：

1. UI 创建和关闭
2. 文本设置
3. 实体生成
4. 动画控制
5. 日志输出
6. 舞台切换
7. 表格读取
8. 时间、输入、调试辅助

#### 特别注意关卡的读取

以“热更切换关卡”为例：

1. 更改\Assets\Resources\Configs下的stageconfig文件

   ![image-20260703170437484](images/hotfix/image-20260703170437484.png)

   关键是要修改其中的“scene字段”，这个字段需要和引擎中的关卡命名一致（关卡id可以自己定义）

   ![image-20260703170717741](images/hotfix/image-20260703170717741.png)

   然后通过await ScriptRuntime.TryReplaceStage(Id)去切换关卡

## 4. 示例项目热更层职责分配说明

这一节按当前 `HotfixProject~` 里实际存在的主要文件展开，帮助快速判断“代码应该落在哪个文件”。

### 4.1 入口与总流程

#### `Entry.cs`

职责：

1. 作为 ILRuntime 入口类型 `HotFix.Entry`
2. 防止热更重复启动
3. 异步拉起 `KongFuGame.Start()`
4. 在 `Tick()` 中依次驱动事件分发、示例逻辑和主业务 Tick
5. 在 `FixedTick()` 中驱动 Trigger 事件和主业务 FixedTick
6. 在 `Stop()` 时统一清理事件和业务状态

#### `KongFuGame.cs`

职责：

1. 管理热更业务整体启动顺序
2. 在进入大厅前先加载关键表和配置
3. 维护“是否已经进入战斗”的主流程状态
4. 负责大厅和战斗之间的切换
5. 转发 `Tick()` / `FixedTick()` 到战斗主流程
6. 在 `Reset()` 中收束全局业务状态

#### `KF_StageManager.cs`

职责：

1. 定义当前项目热更侧使用的舞台 ID
2. 记录当前热更侧认为的舞台状态
3. 封装 `ScriptRuntime.TryReplaceStage(...)`
4. 用 `IsSwitching` 防止并发切换舞台

### 4.2 资源与配置

#### `KF_ResourceManager.cs`

职责：

1. 负责热更 JSON 配置加载
2. 通过 `KongfuRuntimeApi.TryLoadHotfixConfigObject(...)` 获取宿主反序列化结果
3. 缓存业务配置对象
4. 对外提供更贴近业务语义的读取函数

#### `KF_ResourceManager1.cs`

职责：

1. 统一加载 `.te` 表
2. 把大量表数据转成运行期缓存
3. 对外暴露高层查询接口
4. 承担当前项目主要的表数据访问封装

### 4.3 页面与局外 UI

#### `KF_LobbyUIManager.cs`

职责：

1. 拉起大厅主界面
2. 管理大厅主要交互流程
3. 维护局外通用资源显示
4. 负责从大厅跳转到角色、秘籍、勾玉、战斗等入口

#### `KF_CharacterUIManager.cs`

职责：

1. 维护角色系统主 UI
2. 维护兵器、淬炼、奖励等养成相关页面
3. 管理角色展示实体和动画预览
4. 负责角色成长相关交互

#### `KF_SecretMenualUIManager.cs`

职责：

1. 管理秘籍系统主界面
2. 负责秘籍抽取、选择、说明、卡面展示
3. 维护秘籍槽位和购买流程

#### `KF_MagatamaUIManager.cs`

职责：

1. 管理勾玉系统主界面
2. 处理勾玉升级交互
3. 维护勾玉资源消耗与展示

#### `KF_BattleUIManager.cs`

职责：

1. 管理战斗内 UI 主界面
2. 管理战斗加载页、暂停、结算、资源展示等界面
3. 承接战斗流程与 UI 表现层之间的适配

### 4.4 战斗相关文件

#### `Battle/KF_BattleState.cs`

职责：

1. 战斗总状态机
2. 战斗开始前准备流程
3. 对象池预热
4. 战斗速度、暂停、波次控制
5. 战斗主循环和定帧驱动

#### `Battle/KF_AbilityManager.cs`

职责：

1. 管理技能和能力系统数据
2. 负责技能升级与技能效果应用
3. 管理技能运行中的表现和结算

#### `Battle/KF_PlayerWaveAgent.cs`

职责：

1. 管理波阵、阵法、勾玉在战斗中的表现

职责：

#### `Battle/KF_AttackRangeIndicator.cs`

职责：

1. 管理玩家攻击范围指示器表现

#### `Battle/KF_BattleState.cs` 与其他战斗文件的协作关系

职责：

### 4.5 事件桥接层

#### `Framework/Events/UISignalDispatcher.cs`

职责：

1. 用 `listenerId` 方式注册 UI 信号监听
2. 每帧轮询宿主层 UI 信号队列
3. 把宿主信号转发成热更 Action
4. 支持按窗口、按监听器移除绑定

适合放进这里的内容：

1. UI 信号注册和分发基础设施
2. 与具体页面无关的统一监听管理

不适合：

1. 某个具体按钮点击后的业务逻辑
2. 页面状态数据本身

#### `Framework/Events/AnimationEventDispatcher.cs`

职责：

1. 统一消费宿主层动画事件
2. 把动画事件转发给热更注册方

适合放进这里的内容：

1. 动画事件派发基础设施
2. 动画事件监听生命周期管理

#### `Framework/Events/Trigger2DDispatcher.cs`

职责：

1. 统一消费 2D Trigger 事件
2. 通过轮询方式给热更层提供碰撞事件

适合放进这里的内容：

1. Trigger 注册
2. Trigger 轮询消费
3. Trigger 解绑和清理

这一层的定位要始终记住：

它是“事件适配层”，不是“玩法层”。
